# Localization model — language-agnostic internals, content as locale modules

> **STATUS: APPLIED 2026-06-14 (Option A).** `localized_text` is live (migration 004); all authored content
> (6,937 rows) moved to it under locale `pt-BR`; exporters/persisters/validators use neutral field names via
> `scripts/ingest/i18n_text.py`. The legacy `*_pt` columns are **vestigial (NULLed), deprecated** — a future
> migration may drop them; fresh rebuilds leave them empty and use `localized_text`. Adding a locale now = new
> rows, no schema change.


> Owner directive (2026-06-14): the **internal layer** (schema, identifiers, file/dir names, code, enum/type
> values, keys) is **English and language-neutral**. Only **content** (translations, meanings, explanations,
> lesson text, labels, names, descriptions) is localized. **Portuguese (pt-BR) is the only locale for now**,
> but it is a **locale module** — more languages can be added later with **no structural change**.
> **Plan now; apply as a PRE-P5 task** (before mass production, so we don't generate thousands of rows in a
> PT-bound shape and then have to migrate them).

## Principles
- **Identifiers / enums / keys → English, neutral.** e.g. `<note type="l1-pitfall">` (not `armadilha-pt`);
  "L1" = the learner's native language (currently pt-BR), never hard-coded as "pt" in structure.
- **Content → localized**, conceptually keyed by a `locale` tag (`pt-BR` today; `es-LA`, `en-US`… later).
- **Locale dimension, not a schema axis.** Adding a language = adding content rows for that locale, never a
  schema change — the same future-proofing as the corpus's level-agnostic rule (§1.6).

## Current state (PT-bound — to migrate)
The corpus schema currently uses PT-suffixed columns that are really "the pt-BR locale":
`meanings_pt` (kanji), `gloss_pt`/`notes_pt` (vocab senses), `label_pt`/`explanation_pt`/`formation_pt`/
`nuance_pt` (grammar), `pt`/`pt_literal`/`structure_explanation_pt` (sentence), `role_pt`/`gloss_pt`/
`conjugation_note_pt` (token), `function_pt`/`explanation_pt` (particle), family
`label_pt`/`description_pt`/`governing_rule_pt`, topic `objectives_pt`, module `overview_pt`, lesson `body`/
objectives. The lesson format's `<note>` enum has already been de-PT'd (`l1-advantage`/`l1-pitfall`).

## Migration options (pick one in the pre-P5 task)
- **A — `localized_text` table (recommended):** `(entity_type, entity_id, field, locale) → text`. Registries
  keep language-neutral fields; PT rows seeded from the current `_pt` columns; app/exports join by active
  locale. Cleanest for many languages.
- **B — neutralize columns in place:** rename `_pt` → neutral field names with a `locale` column / per-locale
  rows, or store each field as JSON `{locale: value}`. Lighter, less normalized.

Write the migration, move existing PT content into the locale form, re-export. Identifiers, file names, and
exports stay English; only the text values are locale-scoped.

## Why pre-P5
Doing it now delays the heavy lift; doing it after bulk generation means migrating thousands of rows. So the
**first task when mass production resumes** is this migration — then all P5/P6 content lands locale-aware.
